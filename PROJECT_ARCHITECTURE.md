# Архитектура проекта StardewMC

Этот документ описывает текущую архитектуру проекта для навигации и разработки. Проект представляет собой модификацию для Minecraft (Forge), реализующую механики Stardew Valley.

**Root Package:** `dev.flomik.stardew`
**Mod ID:** `stardew`

## 1. Структура Пакетов

Проект разделен на логические слои, где основным принципом является модульность.

### `dev.flomik.stardew`
*   `StardewMod.java`: Точка входа в мод. Инициализация реестров, загрузка конфигов, слушатели событий жизненного цикла.
*   **`client`**: Клиентская логика.
    *   `renderer`: Рендереры для BlockEntity (сундуки и т.д.).
    *   `screen`: GUI экраны (меню).
    *   `model`: Модели блоков и предметов.
    *   `ClientSetup.java`: Регистрация рендереров и слоев.
*   **`common`**: Общая логика (сервер + клиент).
    *   **`module`**: **Основная логика разбита на модули**. Каждый аспект игры находится в своем подпакете:
        *   `farming`: Культуры, грядки, удобрения.
        *   `machinery`: Механизмы (бочонки, прессы, сундуки).
        *   `nature`: Природные блоки (деревья, мусор).
        *   `player`: Логика игрока (сон, события).
        *   `time`: Система времени и сезонов.
        *   `tools`: Инструменты (лейки, мотыги).
    *   `registry`: Централизованная регистрация контента.
        *   `framework`: **DSL для регистрации** (см. ниже).
    *   `api`: Интерфейсы для взаимодействия (Quality, Shape).
    *   `admin`: Команды для отладки (смена сезона, даты).
*   **`core`**: Ядро и утилиты.
    *   `config`: Конфигурация мода.
    *   `network`: Сетевые пакеты (`PacketHandler`).
*   **`mixin`**: Изменения ванильной логики (в основном Stack Size).

## 2. Принципы Разработки

### 2.1. Модульность
Вся игровая логика должна находиться в `common/module/{название_модуля}`. Не следует сваливать всё в один пакет. Если добавляется новая механика (например, NPC), она должна быть в `common/module/npc`.

### 2.2. Регистрация через DSL (Builder Pattern)
В проекте используется кастомный фреймворк для регистрации блоков и предметов, расположенный в `common/registry/framework`.
Вместо стандартного `DeferredRegister.register`, используется `BlockBuilder`.

**Пример использования (`ModBlocks.java`):**
```java
public static final BlockEntry<BlockCrop, Item> PARSNIP_CROP = BlockBuilder
    .create("parsnip_crop", BlockCrop::new) // Фабрика блока
    .initialProperties(() -> BlockBehaviour.Properties.copy(Blocks.WHEAT)) // Свойства
    .noItem() // Блок без ItemBlock (для культур)
    .register(); // Финализация
```

**Возможности `BlockBuilder`:**
*   `.initialProperties(...)` / `.properties(...)`: Настройка свойств блока.
*   `.seasonal(boolean)`: Пометка блока как сезонного.
*   `.blockEntity(...)`: Привязка BlockEntity.
*   `.visual(...)`: Привязка генератора модели (для Datagen).
*   `.item()`: Настройка ItemBlock (табы, стаки).

### 2.3. Data Generation (Datagen)
Генерация данных (модели, лут-таблицы) интегрирована в процесс регистрации.
Используется `DataGenManager` и `ItemModelGen`.
При регистрации предмета/блока можно указать `.visual(ModelPresets.generated(...))`, чтобы автоматически создать JSON модели при запуске DataGen.

### 2.4. Клиент-Сервер разделение
*   Весь рендеринг строго в `client` пакете.
*   Сетевые пакеты регистрируются в `core/network/PacketHandler.java`.

## 3. Ключевые Системы

### 3.1. Система Времени и Сезонов (`common/module/time`)
Мод использует свою абстракцию времени поверх ванильных тиков.
*   **Seasons**: `SPRING`, `SUMMER`, `FALL`, `WINTER`.
*   **Date**: Хранится в `StardewDateData` (World Saved Data).
*   **ScheduleManager**: Позволяет регистрировать задачи на определенное время (например, 6:00 утра).
*   **Начало дня**: Логическое начало дня в 6:00. Сон перематывает время на 6:00 следующего дня и запускает `MorningPass`.

### 3.2. Stack Size (`mixin/stacksize`)
Размер стаков изменен через миксины для соответствия механике Stardew Valley (или 999/кастомные значения). Логика разбросана по миксинам `ItemMixin`, `ItemStackMixin` и т.д.

### 3.3. Block Entities
Используется стандартный Forge подход. Регистрация через `BlockBuilder.blockEntity(...)`. Рендереры регистрируются автоматически через `.renderer(...)` в билдере, если это указано.

## 4. Подход к реализации NPC (Рекомендации на основе архитектуры)

Для реализации NPC следует придерживаться существующей структуры:
1.  Создать модуль `common/module/npc`.
2.  Использовать `StardewRegistry` для регистрации EntityType.
3.  Интегрироваться с `common/module/time/ScheduleManager` для реализации расписания NPC (передвижение в зависимости от времени).
4.  Для данных NPC использовать Capabilities или SavedData, аналогично `StardewDateData`.
5.  Клиентскую часть (модели, рендер) поместить в `client/renderer/npc` (или подобное).
